## 1. 模块概述

### 1.1 模块定位

Module 6 是 Socrates 系统中的检索增强生成（RAG）知识库模块，承担着为整个 tutoring 系统提供结构化数学知识检索能力的核心职责。该模块将分散的 PDF 知识点文档、方法论文档、题目解析文档通过向量化技术统一存储和管理，使其他模块能够在生成提示词时动态检索相关知识片段，从而提升 AI 生成内容的准确性和专业性。

在 Socrates 的整体架构中，Module 6 属于基础设施层，为 Module 2（干预提示生成器）、Module 3（题目推荐引擎）、Module 5（教学策略选择器）提供知识检索服务。其他模块不再需要依赖硬编码的知识，而是通过实时检索从知识库中获取与当前上下文最相关的知识片段。

### 1.2 为什么需要 Module 6

Socrates 系统的核心目标是提供个性化的高中数学辅导。要实现这一目标，系统必须能够理解和生成与数学知识点深度相关的内容。早期版本的系统面临以下挑战：

第一，知识与提示词分离的问题。Module 2 的提示词生成器在构造推理提示时，无法直接引用具体的知识点定义或解题方法。虽然提示词中可以泛泛地提到"使用数学归纳法"，但缺乏对数学归纳法完整定义的引用，导致生成内容可能存在偏差。

第二，知识更新困难的问题。数学教学内容和方法的更新需要同步修改代码中的提示词模板，这种紧耦合方式既不灵活，也容易出错。需要一个独立的知识管理层来解耦知识内容和使用方式。

第三，推荐系统缺乏语义理解的问题。Module 3 的题目推荐基于简单的标签匹配，无法理解知识点的语义关联。例如，学生在"数列"知识点上有薄弱，但"数列"与"数学归纳法"存在天然联系，现有的标签系统无法捕捉这种深层关系。

Module 6 通过构建向量化的知识库解决了上述问题。知识以向量形式存储，使得语义相似性检索成为可能。知识库与使用模块完全解耦，可以通过更新知识文档来更新系统知识，而无需修改代码。

### 1.3 模块职责边界

Module 6 的核心职责包括三个层面。

在知识管理层面，Module 6 负责接收和解析各类数学教学文档（PDF 格式），将文档内容进行结构化处理后存储到向量数据库中。知识库支持按知识点类型（knowledge_point、method、concept、example、strategy）、年级、难度等维度进行组织和筛选。

在检索服务层面，Module 6 提供基于语义向量的相似性检索能力。调用方可以传入自然语言查询，系统返回与查询语义最相关的知识片段列表。检索结果可按相关性进行排序，并支持按元数据进行过滤。

在与外部模块交互层面，Module 6 作为被调用方存在。它不主动推送任何内容，而是响应其他模块的检索请求。Module 6 提供标准化的服务接口，其他模块通过调用 retrieve 方法获取检索结果。

### 1.4 与现有模块的关系

Socrates 系统已包含五个功能模块。Module 1 负责任务拆解与求解，负责将复杂题目分解为可执行的子步骤。Module 2 负责在学生遇到困难时生成实时干预提示词。Module 3 负责任题推荐，基于学生的能力画像推荐合适的练习题目。Module 4 负责学生画像管理，记录和维护学生的学习历史、能力评估等数据。Module 5 负责教学策略选择，为不同类型的学生选择不同的教学风格和节奏。

Module 6 与上述模块的关系是服务与被服务的关系。Module 2、Module 3、Module 5 在执行其核心功能时，可以通过调用 Module 6 来获取知识检索能力，从而增强各自生成内容的质量。这种增强是可选的，模块可以选择是否接入知识库检索，以及在哪些环节使用检索结果。

## 2. RAG 架构设计

### 2.1 整体架构

Module 6 采用标准的检索增强生成架构，由 ingestion pipeline 和 retrieval service 两大部分组成。数据流向从左侧的 PDF 文档开始，经过 extraction、chunking、embedding、storage 四个处理阶段，最终存储到 ChromaDB 向量数据库中。检索时，查询请求经过 embedding 过程后在向量数据库中进行相似性搜索，返回最匹配的知识片段。

整个架构的设计遵循了数据与逻辑分离的原则。知识文档以向量形式存储在 ChromaDB 中，与业务逻辑完全解耦。当知识内容需要更新时，只需重新运行 ingestion pipeline 更新向量库，业务代码无需修改。

### 2.2 Ingestion Pipeline

Ingestion Pipeline 是将原始 PDF 文档转化为可检索向量数据的处理流水线。该流水线包含四个连续的阶段，每个阶段都有明确的输入输出规范。

**阶段一：PDF 文本提取**

该阶段使用 PDF 解析库从 PDF 文件中提取原始文本内容。PDF 解析面临的主要挑战是数学公式的处理。高中数学文档中包含大量 LaTeX 格式的公式，解析库需要能够正确识别和提取这些公式文本。提取的文本应保持原有的段落结构和章节组织，因为后续的 chunking 阶段需要依赖这些结构信息。

对于扫描版 PDF，需要预先使用 OCR 工具进行文字识别。OCR 识别率直接影响后续处理的质量，因此需要选择支持数学符号识别的 OCR 引擎。

**阶段二：结构化 Chunking**

该阶段将提取的文本按语义边界切分为独立的知识片段。Chunking 策略直接影响检索质量，切分粒度太粗会导致检索结果包含过多无关内容，切分粒度太细则会丢失完整的语义信息。

本模块采用基于文档结构的自适应 chunking 策略。首先按章节标题进行一级切分，然后在章节内部按段落进行二级切分。每个 chunk 的目标长度控制在 300 到 500 个中文字符之间，这个长度既能保留完整的语义单元，又不会过于冗长。

对于包含多个知识点的长文档，系统会识别文档中的知识点标记（如"定义"、"定理"、"例题"等），以这些标记为边界进行切分，确保每个 chunk 对应一个独立的知识单元。

**阶段三：向量嵌入**

该阶段使用 DashScope 的 qwen-embeddings-v1 模型将文本 chunk 转换为 1024 维的稠密向量。qwen-embeddings-v1 是阿里云自研的文本嵌入模型，在中文语义理解任务上表现优异，特别适合数学这类专业性强的领域。

每个 chunk 独立调用一次 embedding API，生成对应的向量表示。API 调用采用批量模式以提高效率，单次批量处理的 chunk 数量不超过 32 条。向量生成后与原始文本 chunk 一并传递给下一个阶段。

**阶段四：向量存储**

该阶段将文本 chunk 及其对应的向量、元数据一并写入 ChromaDB 向量数据库。ChromaDB 的 collection 命名空间用于隔离不同类型的知识内容。本模块创建的 collection 名称为 "math_knowledge"。

写入操作支持幂等性，即相同内容的文档多次写入不会产生重复记录。系统使用文档内容的 SHA256 哈希值作为去重依据，相同哈希值的文档会被更新而非重复创建。

### 2.3 Retrieval Service

RAGService 是 Module 6 对外提供服务的核心类，封装了所有检索相关的业务逻辑。外部模块通过调用 RAGService 的 retrieve 方法来获取知识检索能力。

**检索流程**

retrieve 方法接收三个参数：query（查询字符串）、top_k（返回结果数量，默认为 3）、filter（过滤条件，可选）。方法的返回类型为 List[Chunk]，每个 Chunk 包含文本内容、元数据和相关性分数。

检索流程分为五个步骤。首先，对输入的 query 文本调用 DashScope embedding API 生成查询向量。然后，在 ChromaDB 中执行向量相似性搜索，使用余弦相似度作为度量标准，返回 top_k 个最相似的文档。如果提供了 filter 参数，系统会在向量搜索结果基础上进行元数据过滤。

接下来，系统对检索结果进行后处理，按照相似度分数从高到低排序，并添加检索来源标注。最后，返回处理后的 Chunk 列表给调用方。

**过滤机制**

filter 参数支持按以下维度进行精确过滤：type 字段支持按文档类型过滤，可选值为 knowledge_point、method、concept、example、strategy；grade 字段支持按年级过滤，如 high_school；difficulty 字段支持按难度过滤，可选值为 easy、medium、hard；keywords 字段支持按关键词模糊匹配。

过滤操作在向量检索之后执行，这种设计的好处是既能利用向量检索的语义相似性能力，又能通过元数据过滤确保结果的精确性。

### 2.4 技术栈选型

**ChromaDB 向量数据库**

ChromaDB 是本模块选用的向量数据库，作为 Chroma 公司的开源产品，它提供了简洁的 API 和良好的 Python 原生支持。ChromaDB 支持本地持久化存储，数据完全保存在本地文件系统，不依赖外部数据库服务，这与 Socrates 系统轻量级部署的要求高度契合。

相比 Pinecone、Weaviate 等云端向量数据库服务，ChromaDB 的本地部署方式避免了网络延迟和外部依赖，数据安全性也更有保障。ChromaDB 的性能对于 Socrates 的规模来说是足够的——系统初期预计存储不超过 10 万个知识片段，在这个量级下 ChromaDB 的检索延迟可以控制在毫秒级别。

**DashScope qwen-embeddings-v1 嵌入模型**

qwen-embeddings-v1 是阿里云 DashScope 平台提供的文本嵌入模型，输出维度为 1024。该模型在中文语言理解任务上经过了充分训练，对数学领域的专业术语有较好的理解能力。

DashScope 提供 RESTful API 接口，本模块通过 HTTP 请求调用嵌入服务。API 调用采用按量计费模式，对于 Socrates 系统的使用规模（日均调用量预计不超过 1000 次），成本可以忽略不计。

## 3. 数据模型

### 3.1 ChromaDB Collection Schema

本模块在 ChromaDB 中创建名为 "math_knowledge" 的 collection。Collection 的 schema 设计需要平衡检索灵活性与存储效率。

| 字段名    | 类型        | 说明                                                                                         |
| --------- | ----------- | -------------------------------------------------------------------------------------------- |
| id        | string      | 文档唯一标识符，格式为 "{type}\_{序号}"，如 "kp_001"                                         |
| embedding | float[1024] | DashScope qwen-embeddings-v1 生成的文本向量                                                  |
| document  | string      | 文档原始文本内容，即知识片段的完整表述                                                       |
| metadata  | object      | 元数据对象，包含 type、name、keywords、grade、difficulty、related_kp、related_methods 等字段 |

metadata 对象的详细字段定义如下：

```json
{
  "type": "knowledge_point | method | concept | example | strategy",
  "name": "知识片段的名称或标题",
  "keywords": ["关键词1", "关键词2", "关键词3"],
  "grade": "high_school",
  "difficulty": "easy | medium | hard",
  "related_kp": ["相关知识点1", "相关知识点2"],
  "related_methods": ["相关方法1", "相关方法2"]
}
```

### 3.2 文档类型定义

系统支持五种类型的知识文档，每种类型有其特定的使用场景和结构规范。

**knowledge_point 类型**

knowledge_point 是知识库中最核心的文档类型，代表高中数学的独立知识点。每个知识点文档应包含该知识点的定义、核心性质、常见应用场景三个部分。定义部分给出知识点的严格数学定义；性质部分列出该知识点具有的重要性质和定理；应用部分说明该知识点在解题中的常见用法。

示例文档结构：

```
知识点名称：数学归纳法

定义：数学归纳法是证明与正整数n有关的命题的一种方法。

性质：
1. 基本步：证明当n=1时命题成立
2. 归纳步：假设当n=k时命题成立，证明当n=k+1时命题也成立
3. 结论：从而对所有正整数n，命题都成立

应用场景：
- 证明数列通项公式
- 证明整除性命题
- 证明不等式
```

**method 类型**

method 文档记录解题方法和技巧。相比知识点，方法文档更注重操作性，通常包含方法的使用步骤、适用条件、注意事项。方法文档的命名应简洁明确，如"换元法"、"配方法"、"构造辅助函数"等。

**concept 类型**

concept 文档用于解释数学概念。与知识点的区别在于，概念更基础、更抽象，通常是定义其他知识点的前置知识。例如"函数"、"变量"、"定义域"等属于概念类型。

**example 类型**

example 文档记录典型例题及其详细解答。每个例题文档应包含题目描述、思路分析、详细解答、关键点点评四个部分。例题文档的关键词应包含其考察的知识点标签，便于推荐系统进行匹配。

**strategy 类型**

strategy 文档描述教学策略说明。这些文档不是面向学生的知识内容，而是面向系统如何进行教学指导的元信息。例如"如何引导学生发现错误"、"探究式教学的实施步骤"等。

### 3.3 元数据设计考量

元数据的设计需要考虑未来扩展的灵活性，同时保持与现有模块的数据格式兼容性。

keywords 字段采用数组格式，支持多个关键词的关联。这与 Module 3 的标签匹配逻辑兼容，推荐系统可以将学生的知识点薄弱项转换为关键词列表进行检索。

related_kp 和 related_methods 字段建立了知识点之间的关联网络。这个网络可以用于两个方面：一是支持 Module 3 的关联推荐，当学生在某个知识点上有薄弱时，可以推荐与该知识点相关的其他知识点进行巩固练习；二是支持 Module 6 自身的扩展检索，当用户检索某个知识点时，可以同时返回与之相关的其他知识点作为参考。

grade 字段统一使用 "high_school" 值，因为 Socrates 系统目前仅面向高中数学教学。设计为字符串类型是为了未来扩展到初中或大学数学时可以有明确的版本区分。

## 4. 核心功能

### 4.1 Ingestion 功能

**PDF 解析与文本提取**

Ingestion 功能的入口是一个命令行工具，允许运维人员指定 PDF 文件路径来触发文档导入流程。CLI 工具支持单文件导入和批量导入两种模式。

单文件导入命令格式为：`python -m module6.ingestion --file /path/to/document.pdf --type knowledge_point`

批量导入模式需要指定包含多个 PDF 文件的目录路径：`python -m module6.ingestion --dir /path/to/math_knowledge/ --type knowledge_point`

批量导入时，系统会遍历目录下的所有 PDF 文件，依次进行处理。文件名中包含特定关键词（如"例题"、"方法"等）的文件会被自动识别并设置对应的文档类型。

**文本预处理**

提取的文本需要经过清洗和规范化处理。处理步骤包括：移除页眉页脚等干扰信息；将换行符统一为空格或特定分隔标记；处理连续空白字符；识别并标记数学公式段落。

对于 LaTeX 格式的数学公式，系统保留其原始格式而不进行转换。这是因为在后续的提示词生成场景中，LaTeX 格式的公式可以直接被大语言模型理解和使用。

**结构化 Chunking**

Chunking 策略的核心是识别文档中的语义边界。系统采用以下优先级进行切分：

最高优先级是章节标题边界。如果文档中包含明确的章节编号（如"第一章"、"1.1"）或标题样式（加粗、居中），系统会在这些位置进行切分。

中等优先级是段落边界。如果某个段落的长度超过预设的最大 chunk 长度（500 字符），系统会在段落内部继续按句子进行切分。

最低优先级是句子边界。当需要在段落内部进行切分时，系统以完整的句子为最小切分单元，避免出现句子被截断的情况。

每个切分出的 chunk 都会附带其在原始文档中的位置信息，便于后续追溯和更新。

**向量生成与存储**

每个处理完成的 chunk 都会被发送到 DashScope embedding API 进行向量化。API 调用采用异步模式以提高吞吐量，处理结果通过回调机制写入 ChromaDB。

存储时，系统首先生成文档的 SHA256 哈希值作为唯一标识符。如果数据库中已存在相同哈希值的文档，系统执行更新操作而非重复创建。更新操作保留原有的 id，但更新 embedding 向量、文档内容和元数据。

### 4.2 Retrieval 功能

**RAGService.retrieve 方法**

retrieve 是 Module 6 对外暴露的核心检索接口。该方法的签名设计如下：

```python
def retrieve(
    query: str,
    top_k: int = 3,
    filter: Optional[dict] = None
) -> List[Chunk]
```

参数说明：query 是自然语言查询字符串，可以是一个问题、一个知识点名称或一段描述性文本。top_k 指定返回结果的数量，默认值为 3，根据经验，3 个结果能够提供足够的上下文同时不会产生信息冗余。filter 是可选的过滤条件字典，支持按元数据字段进行精确匹配。

返回的 Chunk 对象包含以下属性：text 表示检索到的知识片段文本；metadata 表示该片段的完整元数据；score 表示与查询的相似度分数，取值范围为 0 到 1 之间的浮点数。

**语义检索原理**

检索过程利用了深度学习模型的语义理解能力。当用户输入"如何使用数学归纳法证明数列问题"时，系统首先将该文本转换为 1024 维的向量表示。这个向量编码了查询内容的语义信息，使得语义相似的内容在向量空间中具有相近的位置。

ChromaDB 在检索时计算查询向量与数据库中所有文档向量的余弦相似度，返回相似度最高的 top_k 个结果。这种检索方式不依赖关键词匹配，因此能够捕捉查询和文档之间的语义关联。

**结果后处理**

检索结果在返回前会经过一系列后处理步骤。首先是去重处理，如果多个检索结果的内容相似度超过 0.95，系统会合并这些结果，避免向调用方返回高度重复的内容。

其次是排序优化。向量数据库返回的结果已按相似度排序，但系统会根据文档类型进行微调。例如，当查询是"什么是函数"时，concept 类型文档的优先级会略微提高。

最后是来源标注。每个返回的 Chunk 都会附带其来源文档的信息，包括原始 PDF 文件名、在文档中的位置等，便于调用方进行溯源。

### 4.3 知识库管理功能

**文档更新机制**

当知识内容发生变化时，运维人员可以通过 ingestion 工具重新导入更新后的 PDF 文件。系统通过文档哈希值判断内容是否发生变化，只更新有变化的文档。

对于需要精确更新的场景（如修正某个知识点的表述），工具支持指定文档 id 进行定向更新：`python -m module6.ingestion --update-id kp_042 --content "新的知识点内容"`

**知识库统计**

管理接口提供知识库的统计信息，包括：各类型文档的总数、文档总数、collection 的存储大小、最后更新时间等。这些信息可以通过管理后台展示，便于运维人员监控知识库状态。

## 5. 使用场景

### 5.1 Module 2 干预提示增强

Module 2 的核心职责是在学生解题遇到困难时生成实时的干预提示。干预提示的质量直接影响学生的学习效果——过于模糊的提示没有帮助，过于直接的提示又会降低学生的思考参与度。

**场景描述**

当 Module 2 监听到学生在一个涉及"数列递推"的 breakpoint 处卡顿时，它需要生成一条针对性的提示。Module 2 原本的提示词模板可能是这样的："当前步骤涉及数列的递推关系，可以考虑使用数学归纳法来验证你的递推公式。"

这条提示存在两个问题：第一，"数学归纳法"是泛泛而谈，没有给出具体的操作指导；第二，提示与学生当前的具体问题之间的关联不够紧密，学生可能不清楚如何将归纳法应用到当前的递推问题中。

**知识检索增强**

引入 Module 6 之后，Module 2 的提示生成流程增加了知识检索环节。当 Module 2 需要生成干预提示时，它会首先构造一个检索查询。检索查询的内容包括：当前断点类型（递推关系）、涉及的知识点名称（数列）、学生的难度评估（从 Module 4 获取）。

检索查询示例：`"数列递推关系的证明方法，如何使用数学归纳法"`

Module 6 收到检索请求后，返回最相关的知识片段。假设检索到三个结果：第一条是"数学归纳法"知识点文档，包含基本步和归纳步的具体步骤说明；第二条是"数列归纳证明"例题文档，包含一个完整的数列通项公式证明过程；第三条是"递推与归纳的区别"概念文档，解释了两种方法的适用场景差异。

Module 2 收到检索结果后，将这些知识片段注入到提示词模板中。生成的提示词变为："当前你遇到的递推关系证明，可以尝试使用数学归纳法。基本步骤是：先验证n=1时命题成立（基本步），然后假设n=k时成立并证明n=k+1时也成立（归纳步）。参考例题：{检索到的例题摘要}。注意区分递推方法（直接推导通项）和归纳方法（验证+假设）的适用场景。"

这样的提示词不仅指出了方法名称，还提供了具体的操作步骤和参考案例，学生可以根据提示自行完成证明过程。

### 5.2 Module 3 题目推荐增强

Module 3 的核心职责是根据学生的能力画像推荐合适的练习题目。传统的推荐逻辑基于标签匹配——如果学生某个知识点的掌握度低，就推荐考察该知识点的题目。这种逻辑存在覆盖率不足的问题，无法捕捉知识点之间的关联关系。

**场景描述**

假设 Module 4 记录显示学生"数学归纳法"的掌握度为 0.3（满分 1.0），属于薄弱知识点。Module 3 原本的推荐逻辑会直接推荐标注了"数学归纳法"标签的题目。

这个推荐策略存在疏漏。数学归纳法的掌握需要依赖于"数列"和"递推"基础知识的支撑。如果学生在这两个前置知识点上也有欠缺，单纯推荐归纳法题目可能超出其当前能力范围，导致学习挫败感。

**知识检索增强**

引入 Module 6 之后，Module 3 的推荐逻辑增加了知识图谱分析环节。Module 3 首先向 Module 6 查询"数学归纳法"的相关知识点信息，检索条件为：type=knowledge_point，name=数学归纳法。

Module 6 返回的 related_kp 字段显示，数学归纳法与以下知识点存在关联：数列极限、递推关系、直接证明、反证法。Module 3 获取这些关联信息后，可以进一步查询 Module 4 获取学生在这些关联知识点上的掌握情况。

假设查询结果显示，学生在"递推关系"上的掌握度为 0.5，也处于较低水平。Module 3 调整推荐策略为：首先推荐"递推关系"的巩固题目，待学生在此知识点上有进步后，再推荐"数学归纳法"的提升题目。这种分步骤的推荐方式更符合学生的认知规律。

**推荐结果优化**

Module 3 还可以利用知识库中的 example 文档来优化推荐结果。在推荐某道数列题目之前，Module 3 可以先检索类似的典型例题及其解答思路，在推荐结果中一并呈现给学生。这种"例题+练习"的组合推荐方式比单纯推送题目更具教学价值。

### 5.3 Module 5 教学策略增强

Module 5 的核心职责是为不同特征的学生选择合适的教学策略。教学策略的选择需要考虑学生的认知风格、学习偏好、当前能力水平等多维度因素。

**场景描述**

Module 5 需要为一名倾向于视觉学习的学生制定教学策略。Module 5 原本的策略库中包含针对不同学习风格的策略描述，但这些策略是通用的，缺乏与具体数学知识的结合。

当这名学生正在学习"立体几何"相关内容时，Module 5 需要生成针对性的策略建议。如何将抽象的几何概念转化为直观的视觉体验？如何利用空间想象力来辅助理解？这些具体问题需要有专业数学教学知识的支撑。

**知识检索增强**

引入 Module 6 之后，Module 5 在选择教学策略时增加了一步知识检索。检索查询包括：当前学习内容（立体几何）、学习风格（视觉型）、教学环节（概念引入）。

Module 6 返回的知识片段包括："立体几何的空间想象力培养方法"（method 类型）、"几何直观在高中数学中的应用"（concept 类型）、"立体几何教学图示案例"（example 类型）。

Module 5 收到这些知识片段后，将其与策略库中的通用策略进行融合。对于视觉型学习者学习立体几何的场景，生成的策略建议为：优先使用几何模型和动态图示来展示空间关系；引导学生观察实际物体（如教室、书本）来建立空间概念；使用截面动画来展示立体图形的内部结构。

## 6. MVP 规划

### 6.1 MVP 范围定义

MVP（最小可行产品）阶段的目標是在最短时间内构建出可演示的 RAG 知识库系统。MVP 的设计原则是：功能足够验证核心价值，不过度追求完善性。

**MVP 功能清单**

第一项是 PDF ingestion CLI 脚本。该脚本支持读取 PDF 文件、提取文本内容、按段落进行 chunking、调用 DashScope API 生成向量、存储到 ChromaDB。MVP 阶段的脚本为单文件 Python 脚本，不追求参数化的灵活性，专注于正确处理标准格式的 PDF 文件。

第二项是 ChromaDB 环境搭建。创建名为 "math_knowledge" 的 collection，配置持久化存储路径。MVP 阶段不提供 collection 的管理接口，collection 的创建和初始化通过代码完成。

第三项是 RAGService 的 retrieve 方法。实现基本的检索功能，支持按查询文本返回相似文档。top_k 参数默认为 3，filter 参数支持按 type 和 difficulty 过滤。MVP 阶段不处理 filter 中未定义的字段。

第四项是 Module 2 集成。这是 MVP 的核心演示场景。Module 2 在生成干预提示时，调用 Module 6 的 retrieve 方法获取相关知识片段，并将检索结果注入到提示词中。Module 2 的改动限制在提示词模板层面，不修改核心推理逻辑。

第五项是初始知识库内容。MVP 阶段需要准备一份包含 20 到 30 个知识点的初始知识库，内容覆盖高中数学的核心章节（函数、数列、几何等）。这些知识点的来源是经过整理的 Markdown 文档，通过转换脚本生成 PDF 格式后导入。

**MVP 交付物**

MVP 阶段交付的代码产物包括：module6 包目录，包含 ingestion.py（导入脚本）、service.py（RAG 服务）、models.py（数据模型）；初始知识库文件，包含用于 ingestion 的 PDF 文档集合；部署配置文件，包含 ChromaDB 持久化路径配置、DashScope API 密钥配置。

非代码交付物包括：MVP 演示视频，展示 Module 2 提示增强的效果；模块接口文档，说明 retrieve 方法的调用方式；MVP 回顾报告，总结经验教训和 phase 2 改进计划。

### 6.2 Phase 2 扩展规划

Phase 2 的目标是在 MVP 基础上扩展功能覆盖范围，提升系统的自动化程度和智能化水平。

**Module 3 集成**

Phase 2 将实现 Module 3 与 Module 6 的集成。Module 3 在推荐题目之前，先调用 Module 6 查询目标知识点的关联知识点信息，再结合 Module 4 的学生画像数据进行综合推荐。

**Module 5 集成**

Phase 2 将实现 Module 5 与 Module 6 的集成。Module 5 在选择教学策略时，调用 Module 6 检索与当前教学内容相关的教学方法知识，使策略建议更加专业化。

**自动 ingestion 触发**

Phase 2 将实现知识库内容的自动更新机制。当 Module 4 检测到题目库（problem_bank）发生结构性变化（如新增大量某类题目）时，自动触发 ingestion pipeline 重新构建知识库。

**学生画像感知检索**

Phase 2 将增强检索功能的学生个性化能力。retrieve 方法可以接收学生 ID 参数，系统自动从 Module 4 获取学生的知识点掌握情况，在检索时优先返回学生薄弱知识点的相关内容。

**知识库管理后台**

Phase 2 将开发简单的 Web 管理界面，用于查看知识库统计信息、上传新的 PDF 文档、监控检索调用量等。管理后台不追求复杂功能，专注于提升运维效率。

### 6.3 排期规划

MVP 阶段的开发周期预计为 3 周。

第一周为准备阶段，主要任务包括：搭建 ChromaDB 环境和配置持久化存储；准备初始知识库的 PDF 文档；实现 PDF 文本提取和 chunking 逻辑。

第二周为核心开发阶段，主要任务包括：实现 DashScope embedding API 集成；完成 ChromaDB 写入逻辑；开发 RAGService.retrieve 方法；Module 2 集成开发和联调。

第三周为验证优化阶段，主要任务包括：端到端功能验证；编写模块接口文档；录制 MVP 演示；修复发现的问题。

Phase 2 的开发周期预计为 4 到 6 周，具体进度取决于 MVP 完成后的反馈和优先级调整。

## 7. 评价指标

### 7.1 检索质量指标

**召回率（Recall）**

召回率衡量检索系统找到相关文档的能力。对于 RAG 知识库系统，召回率可以定义为：对于一组预设的测试查询，人工标注的相关文档中有多少比例被系统返回。

测试集的构建需要领域专家参与。对于每个测试查询（如"如何证明数列通项公式"），专家标注出知识库中真正相关的文档。系统检索后计算有多少标注文档被命中。

MVP 阶段的召回率目标为大于 70%。这个目标意味着系统能够找到大部分相关知识，但仍可能遗漏一些边缘情况。

**精确率（Precision）**

精确率衡量检索系统返回结果的相关性。对于每个测试查询，计算返回的 top_k 文档中有多少是真正相关的。

精确率的评估同样需要人工标注。如果系统返回的前 3 个文档中有 2 个是相关的，则该查询的精确率为 66.7%。

MVP 阶段的精确率目标为大于 80%。这个目标意味着系统返回的结果大部分是有价值的，不会向调用方返回大量无关内容。

**平均检索延迟**

平均检索延迟衡量系统的响应速度。延迟从接收查询请求开始计时，到返回结果为止。

对于实时性要求高的使用场景（如 Module 2 的干预提示生成），检索延迟会直接影响学生体验到的提示生成时间。MVP 阶段的平均延迟目标为小于 500 毫秒。

### 7.2 业务效果指标

**Module 2 提示质量评估**

Module 2 生成的干预提示需要进行人工评估。评估维度包括：提示的准确性（是否正确引用了相关知识点）、提示的有用性（学生是否能够根据提示找到解题方向）、提示的适度性（是否给出了足够的启发而不至于直接给出答案）。

人工评估采用 5 分制量表，由教学专家对一批 Module 2 生成的提示进行打分。MVP 阶段的目标是平均分达到 4.0 以上。

**Module 3 推荐效果评估**

Module 3 推荐题目的准确性需要进行离线评估和在线评估。离线评估使用历史数据，计算推荐命中率（学生最终作答正确的比例）。在线评估通过 A/B 测试，对比启用 Module 6 增强前后的推荐效果差异。

MVP 阶段不强制要求在线评估指标，但需要建立离线评估的基线数据。

**Module 5 策略满意度**

Module 5 生成的策略建议需要进行学生满意度调查。学生通过简短的反馈表单报告策略建议的有用程度。

MVP 阶段的目标是满意度评分达到 4.2 以上（5 分制）。

### 7.3 系统稳定性指标

**服务可用性**

Module 6 作为服务提供方，需要保证一定的可用性。MVP 阶段的目标可用性为 99%（即每月最多 7 小时维护窗口期）。

**API 错误率**

API 错误率定义为返回非正常响应的调用占比。错误包括 API 超时、ChromaDB 连接失败、embedding 服务不可用等。

MVP 阶段的目标错误率小于 1%。

## 8. 与其他模块的接口

### 8.1 Module 2 接口规范

Module 2 是 MVP 阶段重点集成的模块。Module 2 通过调用 Module 6 的 RAGService 来获取知识检索能力。

**接口方法**

```
RAGService.retrieve(query: str, top_k: int = 3, filter: Optional[dict] = None) -> List[Chunk]
```

Module 2 在生成干预提示时，按以下流程调用该接口：

首先，Module 2 构造检索查询字符串。查询内容的来源包括：当前断点类型（如"递推关系"）、涉及的知识点名称（从 breakpoint 元数据获取）、问题的难度等级。

其次，Module 2 指定 filter 参数。filter 应当包含当前知识点类型，如 {"type": "method"} 或 {"type": "example"}。Module 2 可以根据断点类型决定检索哪种类型的知识文档。

最后，Module 2 接收返回的 Chunk 列表。Module 2 的提示生成逻辑将 Chunk 的 text 字段内容插入到提示词模板中的指定位置。

**调用示例**

```python
from module6.service import RAGService

rag_service = RAGService()

# Module 2 在需要生成提示时调用
query = f"{breakpoint_type} {knowledge_point_name} 解题方法"
filter = {"type": "method", "difficulty": student_difficulty}

chunks = rag_service.retrieve(query=query, top_k=3, filter=filter)

# Module 2 将检索结果注入到提示词模板
hint_prompt = build_hint_prompt(breakpoint, chunks)
```

**集成位置**

Module 2 的集成点在 HintGeneratorV2 类中。当 HintGeneratorV2 需要生成提示时，它首先调用 RAGService 检索相关知识，然后将检索结果作为上下文传递给语言模型生成提示。

### 8.2 Module 3 接口规范

Module 3 在 Phase 2 中与 Module 6 进行集成。Module 3 调用 Module 6 的目的有二：一是获取知识点的关联关系，用于优化推荐策略；二是获取相关例题信息，用于丰富推荐内容。

**接口方法**

Module 3 同样使用 RAGService.retrieve 方法。根据不同的使用场景，Module 3 有两种调用模式。

第一种是知识点关联查询。当 Module 3 需要获取某知识点的关联知识点列表时，调用 retrieve 方法并指定 type=knowledge_point，关键词为知识点名称。

```python
# Module 3 获取知识点的关联信息
chunks = rag_service.retrieve(
    query=f"知识点 {target_kp} 相关知识",
    top_k=5,
    filter={"type": "knowledge_point"}
)

# 从检索结果中提取 related_kp 字段
related_kps = []
for chunk in chunks:
    related_kps.extend(chunk.metadata.get("related_kp", []))
```

第二种是推荐题目关联查询。当 Module 3 准备推荐某道具体题目时，可以检索该题目的典型解法参考。

```python
# Module 3 获取题目的参考解法
chunks = rag_service.retrieve(
    query=f"题目 {problem_id} 解题方法",
    top_k=2,
    filter={"type": "example"}
)
```

**推荐增强流程**

Module 3 的推荐流程增强后变为：首先查询目标知识点的关联知识点；然后查询学生在这些关联知识点上的掌握情况；最后综合评估后确定推荐题目列表。

### 8.3 Module 5 接口规范

Module 5 在 Phase 2 中与 Module 6 进行集成。Module 5 调用 Module 6 的目的是获取与当前教学内容相关的教学策略知识，用于丰富策略建议的专业性。

**接口方法**

Module 5 使用 RAGService.retrieve 方法，检索与当前教学情境相关的 strategy 类型文档。

```python
# Module 5 获取教学策略建议
chunks = rag_service.retrieve(
    query=f"针对{learning_style}学习者讲授{topic}的教学策略",
    top_k=3,
    filter={"type": "strategy"}
)
```

**策略生成增强流程**

Module 5 的策略生成流程增强后变为：首先从 Module 4 获取学生的学习风格偏好；然后调用 Module 6 检索适配该学习风格的策略知识；最后将检索结果与通用策略模板进行融合，生成最终的策略建议。

### 8.4 接口通用约定

**服务发现机制**

Module 6 作为 FastAPI 服务运行，通过 HTTP 接口对外提供检索能力。其他模块通过 HTTP 客户端调用 RAGService 的 RESTful 接口。

MVP 阶段采用直连模式，即 Module 2/3/5 在代码中直接实例化 RAGService 的本地代理对象，由代理对象负责 HTTP 通信。这种模式下，Module 6 的服务地址需要在调用方的环境变量中配置。

**错误处理约定**

RAGService 的 retrieve 方法在遇到以下情况时抛出自定义异常：ChromaDB 连接失败时抛出 ChromaDBConnectionError；embedding 服务调用失败时抛出 EmbeddingServiceError；检索超时（默认 5 秒）时抛出 RetrievalTimeoutError。

调用方应当捕获这些异常并进行适当的降级处理。例如，当 Module 2 无法获取检索结果时，可以回退到不依赖知识检索的标准提示生成流程。

**版本兼容约定**

Module 6 的接口遵循语义化版本规范。retrieve 方法的返回类型 Chunk 包含版本字段，标识该结果所依据的知识库版本。当知识库发生重大更新时，版本号递增。调用方可以通过检查版本号来判断是否需要重新处理检索结果。

## 9. 技术选型说明

### 9.1 ChromaDB 选型理由

**为什么选择 ChromaDB**

ChromaDB 是专门为 LLM 应用设计的开源向量数据库，其设计理念与 Socrates 系统的需求高度契合。

从部署复杂度来看，ChromaDB 支持本地持久化存储，不依赖外部服务。Socrates 系统的规模预计在初期不会很大，引入一个独立的向量数据库服务会增加部署和维护负担。ChromaDB 的嵌入即可运行特性降低了系统的整体复杂度。

从 API 友好性来看，ChromaDB 提供简洁的 Python SDK，接口设计直观易懂。对于快速实现 MVP 来说，ChromaDB 的低学习成本是一个重要优势。开发者可以在几个小时内完成从环境搭建到基本功能实现的完整流程。

从功能完整性来看，ChromaDB 虽然轻量，但提供了向量检索所需的核心功能：支持余弦相似度度量、支持元数据过滤、支持集合操作。对于 Socrates 系统的使用场景来说，ChromaDB 的功能是完备的。

**为什么不选择其他方案**

Pinecone、Weaviate 等云端向量数据库服务提供了更强大的托管能力和弹性扩展性，但对于 Socrates 系统的初期规模来说过于重量级。这类服务的成本模型（按存储空间和查询量计费）对于一个学校或教育机构来说是额外的支出。

FAISS 是 Facebook 开发的向量检索库，以性能著称，但它是一个库而非数据库。使用 FAISS 需要自行处理向量存储、索引管理、持久化等逻辑，增加了开发工作量。ChromaDB 在 FAISS 之上封装了这些功能，同时保持了良好的性能。

Milvus 是国产的成熟向量数据库，功能强大且支持分布式部署。其面向大规模数据场景的设计使得部署和运维复杂度较高。对于 Socrates 系统预计的数据规模（不超过 10 万向量），Milvus 显得过于重型。

### 9.2 DashScope Embedding 选型理由

**为什么选择 qwen-embeddings-v1**

qwen-embeddings-v1 是阿里云通义千问系列提供的文本嵌入模型，在中文语义理解任务上经过了充分优化。高中数学知识库的特点是包含大量专业术语和公式符号，模型对中文的理解能力直接影响检索质量。

从中文能力来看，qwen-embeddings-v1 基于大规模中文语料训练，对中文语义的理解优于多数英文模型的翻译版本。在数学领域，模型能够正确理解"归纳"、"递推"、"收敛"等专业术语的语义。

从使用便利性来看，DashScope 提供 RESTful API 接口，通过 HTTP 请求即可调用。SDK 支持 Python 语言，与 Socrates 后端技术栈一致。相比需要本地部署的模型，DashScope API 方式无需占用服务器计算资源。

从成本角度来看，DashScope 的 embedding API 采用按调用量计费模式。Socrates 系统日均调用量预计不超过 1000 次，月度成本在可接受范围内。对于 MVP 阶段的验证性使用，DashScope 还提供免费额度。

**为什么不选择其他方案**

OpenAI 的 text-embedding-ada-002 是业界广泛使用的嵌入模型，但其在中文任务上的表现不如专门优化的中文模型。且调用 OpenAI API 需要考虑网络延迟和服务可用性问题。

Hugging Face 提供了多种开源嵌入模型，如 mteb 系列。这些模型可以本地部署，数据安全性更好，但需要额外的服务器资源来运行模型推理。考虑到 Socrates 系统的规模和成本预算，使用云端 API 是更务实的选择。

### 9.3 为什么不选择 LangChain

LangChain 是当前流行的 LLM 应用开发框架，提供了 RAG 能力的封装。Socrates 系统在设计 Module 6 时评估了 LangChain 的适用性，最终决定不采用。

从架构耦合度来看，LangChain 追求的是端到端的 LLM 应用开发框架，它将 embedding、向量存储、提示词管理、链式调用等多个组件整合在一起。使用 LangChain 意味着与它建立较强的技术依赖关系。

从灵活性和可控性来看，Socrates 系统已经有自己的模块架构（Module 1-5），引入 LangChain 会增加架构的复杂性。Module 6 的需求相对明确和简单——只需要一个可靠的向量存储和检索能力，自己实现这些功能比引入 LangChain 更轻量。

从学习成本来看，LangChain 的 API 设计复杂度和概念模型有一定学习曲线。考虑到 Module 6 的功能和边界已经比较清晰，直接使用 ChromaDB Python SDK 实现是更高效的选择。

## 10. 依赖和限制

### 10.1 系统依赖

**运行时依赖**

Module 6 的运行依赖以下软件环境：Python 版本 3.10 或更高版本；ChromaDB 版本 0.4.x 或更高版本，通过 pip 安装；DashScope Python SDK，通过 pip 安装；PDF 解析依赖 pdfplumber 或 PyMuPDF。

这些依赖在 Socrates 项目的 requirements.txt 中统一管理。Module 6 不引入额外的依赖项，保持依赖列表的精简。

**外部服务依赖**

Module 6 依赖 DashScope 的 embedding API 服务。这意味着在调用 retrieve 方法时，需要有可用的网络连接访问 DashScope 的服务端点。

DashScope API 的可用性目标为 99.9%，但在实际使用中可能出现临时不可用的情况。Module 6 实现了重试机制（最多 3 次）和降级策略（检索失败时返回空结果而非抛出异常），以应对外部服务的临时故障。

**数据依赖**

Module 6 的初始知识库需要预先准备好 PDF 格式的知识点文档。这些文档由项目组内的教学专家准备，不属于代码仓库的一部分。

文档的质量直接影响知识库的检索效果。MVP 阶段需要 20 到 30 篇经过整理的高质量知识点文档作为初始内容。

### 10.2 性能限制

**向量维度限制**

qwen-embeddings-v1 生成的向量固定为 1024 维。ChromaDB 存储这些向量需要占用一定的磁盘空间。按照每向量 4KB 估算（1024 维 × 4 字节），10 万个向量约占 400MB 存储空间。

**检索延迟限制**

retrieve 方法的端到端延迟包含：查询文本 embedding 生成时间（网络 API 调用，约 200-300 毫秒）、ChromaDB 向量检索时间（约 50-100 毫秒）、结果后处理时间（约 10-20 毫秒）。

在正常网络条件下，MVP 阶段的目标是端到端延迟控制在 500 毫秒以内。如果 DashScope API 响应变慢，延迟会相应增加。

**并发限制**

DashScope API 有请求频率限制，MVP 阶段的使用量不会触及限制。但如果系统规模扩大，需要关注 API 的配额管理。

ChromaDB 本地存储的并发读写能力取决于宿主服务器的硬件配置。对于 Socrates 系统的预期负载（单次检索请求的 QPS 预计不超过 10），ChromaDB 可以轻松应对。

### 10.3 知识库内容限制

**覆盖范围限制**

MVP 阶段的初始知识库只覆盖高中数学的核心章节（函数、数列、解析几何、立体几何、概率统计等）。对于边缘知识点或难题变种，知识库可能没有覆盖。

这意味着 Module 6 的检索结果可能返回空列表或质量不高的结果。调用方需要处理好这种情况，进行降级处理。

**内容时效性限制**

知识库的内容基于某一时间点的快照。数学知识本身相对稳定，但教学方法和解题技巧可能随时间演进。Module 6 不提供自动更新机制，需要人工触发 ingestion 流程来更新内容。

**语言限制**

当前版本的 qwen-embeddings-v1 对简体中文的支持最好，对繁体中文、文言文、符号混排文本的支持可能有限。知识库文档应当使用标准的简体中文撰写，避免使用方言或非正式表达。

### 10.4 安全和隐私考量

**数据存储安全**

ChromaDB 的持久化数据保存在本地文件系统，需要确保服务器的文件系统权限配置正确，防止未授权访问。

知识库中可能包含受版权保护的教材内容。在使用和更新知识库时，需要确认内容的授权情况，避免法律风险。

**API 密钥管理**

DashScope API 密钥需要妥善保管，不得硬编码在代码中。MVP 阶段通过环境变量配置密钥。生产环境应当使用密钥管理服务。
